# Documentation Standards

Patterns for documentation at every level: project README, feature READMEs, usage guides, JSDoc/TSDoc, Swagger API docs, and changelogs.

---

## Documentation Hierarchy

```
project-root/
├── README.md                                  # Project-level: what, why, setup, run, test, deploy
├── CHANGELOG.md                               # Auto-generated by semantic-release
├── .github/
│   ├── README.md                              # CI/CD documentation
│   └── pull_request_template.md               # PR template
├── test/
│   └── README.md                              # Testing guide
├── libs/
│   └── feature/
│       ├── README.md                          # Library-level: architecture, data flow, extensibility
│       └── src/
│           └── processors/
│               └── vendor/
│                   └── README.md              # Component-level: specific behavior, overrides, examples
└── src/
    └── auth/
        └── oauth.usage.md                     # Usage guide for a specific service
```

Every layer of the codebase has documentation at the appropriate granularity.

## Project README (`README.md`)

### Required Sections

```markdown
# Project Name

[Badges: CI status, coverage, version]

[Quick links: CI dashboard, architecture diagram, monitoring]

## Description

What this project does, where it fits in the system, key technologies used.

## Initial Setup

1. Install dependencies.
   ```bash
   npm i
   ```
2. Create an environment file from the template.
   ```bash
   cp .env.example .env
   ```

## Running the app

1. Start infrastructure dependencies.
   ```bash
   docker-compose up
   ```
2. Start the app.
   ```bash
   npm run start:dev
   ```

## Testing

```bash
# unit tests
npm run test

# e2e tests
npm run test:e2e

# test coverage
npm run test:cov
```

## Deployment

- Follow [Conventional Commits](https://www.conventionalcommits.org/) for all commit messages.
- Merge to main triggers automatic nonprod deployment.
- Production deployments are manual via GitHub Actions.

## Additional Resources

Links to related documentation, Redis usage guides, architecture docs, etc.
```

### README Principles

- **Start with setup** — a new developer should be productive within 5 minutes.
- **Show commands, not paragraphs** — `bash` blocks are more useful than prose.
- **Link to external docs** — architecture diagrams, dashboards, Postman collections.
- **Keep it current** — outdated READMEs are worse than none.

High-level description with diagram reference.

## Component Documentation

| Component | Purpose | Documentation |
|-----------|---------|---------------|

### Adding a New Component

Step-by-step instructions with code examples:

1. Create the service implementing the interface
2. Create the module
3. Register with the factory
4. Add documentation

## Testing Strategy

### Unit Tests
- What is tested, how to run

### Integration Tests

### Principles

- **Link to sub-READMEs** using a table with arrow links (`[View Details →](path)`).
- **Include a development guide** for adding new components — this is the most valuable section.
- **Include code examples** directly in the README for common patterns.
- **Testing strategy** is part of the feature docs, not just the test directory.

## Usage Guides (`.usage.md`)

For services with non-obvious APIs, create a usage guide:

```markdown
## OAuthService usage

1. Import `OAuthService` in the integration service where you need an oauth access token.

2. Construct `AuthRequest` as needed for the specific oauth implementation.

3. Pass a `LenderUUID` or `IssuerUUID` (to construct the cache key) along with
   the `AuthRequest` object to the `fetchToken` to return an access token.

The service will handle getting a cached token or making a new request
and setting the result to cache.
```

**When to write a usage guide:**
- The service has a multi-step API that isn't obvious from the type signature.
- There's caching, retry, or lifecycle behavior consumers need to understand.
- The service wraps an external system with project-specific conventions.

**Where to place it:** Co-located with the source file (`src/auth/oauth.usage.md`).

## JSDoc / TSDoc

### When to Write Docstrings

| Scenario | Required | Example |
|----------|----------|---------|
| Interface contracts (public API) | Yes | `ILenderService`, `IntegrationProcessor` |
| Custom exception classes | Yes | One-line explaining when thrown |
| Service public methods | Yes | What it does, not how |
| Utility functions | Yes | Purpose, parameters, edge cases |
| Private methods | No | Only if behavior is non-obvious |
| DTOs / config interfaces | Optional | Only for fields with business meaning |

### Format

**Interface methods — full TSDoc:**

```typescript
export interface IntegrationProcessor {
  /**
   * Validates products against a schema before processing.
   * Ensures product data conforms to the expected structure
   * before any transformation logic is applied.
   *
   * @template T - The generic type for product data returned from the integration
   * @template V - The class type to validate against
   * @param products - Array of products to validate
   * @param validationClass - Class constructor defining the validation schema
   * @returns Promise resolving to array of products that passed validation
   */
  validateProducts<T, V extends object>(
    baseplate: Baseplate,
    products: Product<T>[],
    validationClass: ClassConstructor<V>,
  ): Promise<Product<T>[]>;
}
```

**Service methods — concise:**

```typescript
/**
 * Aggregates offers from multiple integrations based on the request
 */
async aggregateOffers<T>(request: AggregatorRequest): Promise<QuestResponse<T>> {
```

**Utility functions — purpose + edge cases:**

```typescript

 * typically found in a price listing. Ex: $12,000.26
 */

```

**Exceptions — one-line:**

```typescript
/**
 * Thrown when we are unable to format the request for the integration
 */

```

**DTO fields — only when business meaning isn't obvious:**

```typescript
export class Applicant {
  /**
   * Cards will always send last 4 of SSN
   */
  @IsString()
  @IsOptional()
  ssn?: string;
}
```

### Anti-Patterns

Do **not** write comments that restate what the code already says:

```typescript
// BAD
/** Returns the name */
getName(): string { return this.name; }

// BAD
// Import the module
import { Module } from '@nestjs/common';

// GOOD — explains WHY, not WHAT
/**
 * Returns a base64 encoded username:password for use with basic auth
 */
export function base64Credentials(identifier: string, password: string): string {
```

## Swagger / API Documentation

### Configuration

Swagger is enabled conditionally (local dev only):

```typescript
// main.ts
if (process.env.HOST_ENV === 'local') {
  const config = new DocumentBuilder()
    .setTitle(API_DOCS_CONFIG.title)
    .setDescription(API_DOCS_CONFIG.description)
    .setVersion(API_DOCS_CONFIG.version)
    .build();

  const document = SwaggerModule.createDocument(app, config);
  SwaggerModule.setup('api', app, document);
}
```

### Compile-Time Plugin

Use the `@nestjs/swagger` plugin in `nest-cli.json` for automatic DTO documentation:

```json
{
  "compilerOptions": {
    "plugins": [
      {
        "name": "@nestjs/swagger",
        "options": {
          "introspectComments": true
        }
      }
    ]
  }
}
```

With `introspectComments: true`, JSDoc comments on DTO properties automatically become Swagger descriptions.

### Controller Decorators

```typescript
@ApiTags('Cards')
@Controller('integration')
export class AggregatorController {

  @Post('offers')
  @ApiOperation({ summary: 'Returns offers from requested integrations' })
  @ApiHeader({ name: 'Authorization', description: 'Bearer token (JWT)', required: true })
  @ApiBody({ type: AggregatorRequest })
  @ApiOkResponse({ description: 'Successful response', type: QuestResponse })
  async create(@Body() request: AggregatorRequest): Promise<QuestResponse<any>> {
```

**Required decorators per endpoint:**
- `@ApiTags` — groups endpoints in Swagger UI
- `@ApiOperation` — summary of what the endpoint does
- `@ApiBody` — request DTO class
- `@ApiOkResponse` / `@ApiCreatedResponse` — success response with DTO
- `@ApiHeader` — for auth or custom headers

## PR Template

```markdown
<!-- Add the ticket number -->
TPG-

## Description
<!-- What's the purpose of this PR? What was your approach? -->

## Testing
<!-- How did you test that your change works? -->

## Checklist
- [ ] Tests
- [ ] Documentation
```

**Principles:**
- Link to a ticket for traceability.
- Explain **why**, not just **what**.
- Describe how you tested — manual steps, new tests, snapshot updates.
- Checklist enforces that tests and docs are part of every PR.

## Changelog

Changelogs are **auto-generated** by `semantic-release` from Conventional Commit messages. Never edit `CHANGELOG.md` manually.

The pipeline:
1. Commits follow `feat:`, `fix:`, `chore:`, etc.
2. `@semantic-release/commit-analyzer` determines version bump.
3. `@semantic-release/release-notes-generator` creates release notes.
4. `@semantic-release/changelog` updates `CHANGELOG.md`.
5. `@semantic-release/git` commits the changelog.
6. `@semantic-release/github` creates a GitHub release.

## Testing Documentation (`test/README.md`)

Every project should have a testing guide that covers:

1. **Testing structure** — where tests live, what directories contain
2. **Test types** — unit vs E2E, when to use each
3. **Conventions** — naming, variable prefixes, AAA vs Given-When-Then
4. **Snapshot testing** — how to set up, update, and review snapshots
5. **Test data management** — factories, mocks, test cases
6. **Running tests** — all commands and CI integration

## CI/CD Documentation (`.github/README.md`)

Document the CI/CD setup:

- What shared workflows are used and where they come from.
- Required GitHub repository variables and secrets.
- How deployment pipelines work (auto vs manual).
- Branch and tag conventions that trigger deployments.
